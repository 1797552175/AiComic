## 二、根因分析

### 问题1：研发驳回时未填写原因（致命）

19个PROTO任务全部被标记"已驳回"，但**输出结果字段为空**。
唯一有记录的是T010，原因是：
> "auth.py 代码存在于 commit 3eaa169，但该 commit 不在当前分支。当前代码中 auth_router 被注释"

这说明**研发有分析，但拒绝填写输出结果**。

### 问题2：研发Bot执行能力不足

根据历史记录分析，研发Bot的 `execute_proto_task()` 函数：
- 收到PROTO任务后只返回"completed"，没有实际干活
- 可能是CrewAI调用失败，或代码生成质量不达标
- 导致大量任务被批量标记为"已驳回"

### 问题3：PM无法追溯驳回原因

由于输出结果为空，PM无法：
- 判断是原型描述不清还是研发执行不力
- 针对性优化原型或改进任务分配
- 与研发对齐验收标准

## 三、解决方案

### 解决方案1（PM侧）：强制要求驳回必须填原因

在 state-sync Skill 中约定：
- 研发驳回任务**必须**填写输出结果（拒绝的详细原因）
- 不填写原因则视为未完成，继续挂在"待领取"状态
- 状态监控Bot应增加检查：驳回但无输出结果→在群里告警

### 解决方案2（研发侧）：研发Bot应重试而非直接驳回

研发Bot收到PROTO任务后：
1. 先评估任务复杂度（看原型文件）
2. 如果无法完成，应在群里@PM说明原因，请求补充信息
3. 只有在明确无法执行的情况下才标记驳回
4. 驳回时必须填写原因

### 解决方案3：区分"驳回"和"待补充"

将"已驳回"拆分为：
- **已驳回（缺信息）**：原型描述不清，需要PM补充
- **已驳回（无法执行）**：技术限制或依赖未满足
- **已驳回（重复）**：与其他任务重叠
