## 一、产品定位与目标

### 1.1 产品定位

AI 创作动态漫是一款端到端的 AI 驱动的动态漫画创作工具。用户输入文字，AI 自动完成从故事脚本、分镜画面到动态视频的全流程，最终输出可分享的动态漫画内容。

**一句话定位：** "用文字讲故事，让 AI 帮你画出来、动起来"

### 1.2 核心目标

- 将动态漫画创作时间从**数天缩短至分钟级**
- 覆盖从"有个想法"到"可发布内容"的完整链路
- 目标用户：独立创作者、IP 孵化团队、泛娱乐内容消费者

## 二、技术架构设计

### 2.1 系统模块划分

```
┌─────────────────────────────────────────────────────────────┐
│                        用户界面层（Web/App）                  │
├──────────┬──────────┬──────────┬──────────┬────────────────┤
│ 故事输入  │ 剧本编辑  │ 角色管理  │ 预览室   │   导出中心     │
├──────────┴──────────┴──────────┴──────────┴────────────────┤
│                       API 网关层                             │
├──────────┬──────────┬──────────┬──────────┬────────────────┤
│StoryEngine│Character │ Visual   │ Audio    │ Export         │
│（故事生成）│ Engine   │ Engine   │ Engine   │ Service        │
│          │（角色一致）│（绘图+动态）│（配音音效）│（封装导出）   │
├──────────┴──────────┴──────────┴──────────┴────────────────┤
│                  AI 模型层（LLM + 绘图 + TTS）               │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 各引擎职责

**Story Engine（故事生成）**
- 接收入口：用户文本（50-500字）
- 出口：结构化分镜脚本（JSON 格式）
- 调用模型：GPT-4 / Claude（大语言模型）
- 核心 prompt 工程：定义故事扩展逻辑、镜头语言规范

**Character Engine（角色一致性）**
- 接收入口：角色参考图（2-3张）+ 分镜脚本中的角色描述
- 出口：带角色 embedding 的分镜数据 + 一致性评分
- 技术方案：LoRA 微调角色 + CLIP 相似度比对
- 角色数据库：存储已确认角色模板，供后续分镜复用

**Visual Engine（视觉生成）**
- 接收入口：分镜画面描述 + 风格参数
- 出口：静态图 → 动态化后的视频片段
- 绘图模型：Stable Diffusion XL / Midjourney API / 自研
- 动态化方案：SadTalker / Dreamix / 可控视频生成模型
- 并行策略：多分镜并行生成，汇合后按顺序组装

**Audio Engine（音频生成）**
- 接收入口：对话文本、场景情绪标签
- 出口：合成音频文件（WAV/MP3）
- TTS 模型：CosyVoice / GPT-SoVITS（支持情感控制）
- BGM 匹配：根据场景标签从音乐库匹配 + AI 生成

**Export Service（导出服务）**
- 接收：视频片段、音频文件、字幕数据
- 出口：MP4 / GIF / 序列帧
- 使用 FFmpeg 进行最终封装
- 支持水印注入（元数据写入）

## 三、数据模型设计

### 3.1 核心实体

**Project（项目）**
```
Project {
  id: UUID
  title: string
  creator_id: UUID
  status: draft | generating | completed | failed
  created_at: timestamp
  updated_at: timestamp
}
```

**Script（剧本）**
```
Script {
  id: UUID
  project_id: UUID
  outline: string        // 故事大纲
  scenes: Scene[]         // 分镜列表
  style_tags: string[]   // 风格标签
}
```

**Scene（分镜）**
```
Scene {
  index: int
  description: string          // 画面描述
  dialogue: string              // 对话/旁白
  characters: CharacterRef[]   // 出现的角色
  visual_prompt: string         // AI 绘图 prompt
  video_url: string?            // 生成的视频 URL
  audio_url: string?            // 生成的音频 URL
  consistency_score: float?      // 角色一致性评分
}
```

**Character（角色）**
```
Character {
  id: UUID
  project_id: UUID
  name: string
  template_images: string[]     // 参考图 URLs
  lora_path: string?            // 微调后的 LoRA 路径
  description: string           // 角色特征描述
}
```

## 五、生成流程设计

### 5.1 同步 vs 异步策略

- **Story Engine**：同步返回（<3 秒）
- **Character Engine**：同步 + 缓存命中（<5 秒）
- **Visual Engine**：异步任务队列（每个分镜独立任务）
- **Audio Engine**：异步（并行于视觉生成）
- **Export Service**：异步（所有分镜就绪后触发）

### 5.2 分阶段生成

```
阶段 1：故事生成（同步）
  User → Story Engine → Script JSON

阶段 2：角色学习（可选，同步）
  Character Engine → 角色 embedding

阶段 3：并行生成（异步）
  ├─ Visual Engine → 各分镜静态图
  ├─ Visual Engine → 各分镜动态化
  └─ Audio Engine → 各分镜配音+BGM

阶段 4：组装导出（异步）
  FFmpeg → 最终 MP4/GIF
```

## 七、迭代路线图

### Phase 1（MVP）：单线故事生成
- 输入一段文字 → 生成 4-8 个分镜
- 静态图输出，角色一致性 70%
- 单人主角，默认风格

### Phase 2（v2.0）：完整链路
- 加入动态化、配音、BGM
- 多角色支持
- 用户可编辑脚本

### Phase 3（v3.0）：协作与社区
- 角色库（社区共享角色）
- 模板市场（预设剧本/风格）
- 团队协作

## 八、风险与挑战

1. **角色一致性**：多分镜角色一致性是技术核心难点，需要持续优化 LoRA + CLIP 方案
2. **生成时长**：完整链路端到端需控制在 5 分钟内，用户体验关键
3. **内容合规**：AI 生成内容需接入审核模块，防止滥用
4. **成本控制**：GPU 算力成本高，需精细化缓存策略
