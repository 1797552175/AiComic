# Full-Load Monitor Skill

## 触发词
- "全员干活"
- "满负载工作"
- "监控全员工作"

## 功能说明
当用户说"全员干活X小时"，Monitor 自动进入实时调度模式：
1. 持续监控所有 Bot（Dev/PM/Marketing）的状态
2. 每 5 秒检查一次，发现空闲立即分配任务
3. 持续监控指定时长

## 使用方式
```
用户: 全员干活2小时
Monitor: 启动实时调度，持续2小时，实时分配任务
```

## ⚠️ 最高优先级：任务完成后必须立即更新任务板

当任何一个 Bot 完成工作时，Monitor **第一件事**就是更新任务板：

### 必须更新的字段
1. **状态** → 根据结果改成：已完成/失败/待合并
2. **分配给** → 谁执行的
3. **依赖任务** → 关联的前置任务
4. **完成时间** → 实际完成时间
5. **输出结果** → 产出文件路径、结果摘要

### 更新顺序（绝对不能变）
```
Bot完成任务 → 1. 更新任务板状态 ← 最高优先级
           → 2. 通知相关Bot
           → 3. 分配下一个任务
```

## 工作优先级

### Dev Bot
1. 执行原型研发任务（6 Agent并行）
2. 优化后端API接口
3. Bug修复和测试
4. 其他研发任务

### PM Bot（优先级从高到低）
1. **最高优先**：任务板原型<5个 → 立马产出新原型
2. **第二优先**：研发驳回的原型 → 优化并更新任务状态
3. **闲时自维护**：原型优化（获客导向）、场景补充、竞品差距分析

### Marketing Bot（优先级从高到低）
1. **最高优先**：分析已完成的任务
   - 分析完了 → 更新任务状态为"待删除"
   - 分析结论"不符合市场" → 向PM提需求
   - 分析结论"符合导向" → 产出营销方案
2. **闲时任务**：
   - 以获客为导向，提项目营销需求
   - 整理项目原型和文档
   - 分析产品原型是否重复或不合理

## 工作原理

### 监控循环
```
while 剩余时间 > 0:
    检查 Dev Bot 状态
    检查 PM Bot 状态  
    检查 Marketing Bot 状态
    
    if Bot 空闲:
        分配任务
        记录日志
    
    sleep(5秒)
```

### Bot 端口
- Dev Bot: 8003
- PM Bot: 8002
- Marketing Bot: 8004
- Monitor: 8001

### 任务分配
- 任务类型: `dev` | `pm` | `marketing`
- API 端点: POST /execute
- Header: `X-API-Key: aicomic-shared-secret-key-2026`

### 状态检查
```bash
curl -s http://localhost:800{2,3,4}/status | python3 -c "import sys,json; print(json.load(sys.stdin)['status'])"
```

## 任务池（按需扩展）

### Dev Bot 任务
- 执行原型研发任务（6 Agent并行开发）
- 优化后端API接口
- 修复Bug并编写测试用例
- 实现图片生成模块
- 优化数据库查询性能
- 编写API文档和示例

### PM Bot 任务
- 优化原型文档（获客导向）- 简化操作流程
- 优化原型文档 - 增加新手引导截图
- 优化原型文档 - 补充真实使用场景
- 评估功能优先级，输出MVP清单
- 分析竞品功能差距，制定差异化策略
- 整理产品需求变更日志

### Marketing Bot 任务
- 优化产品介绍文档（获客导向）
- 生成小红书种草文案（含截图示例）
- 生成知乎技术文章
- 生成公众号推广文章
- 制作产品使用场景案例
- 整理竞品对比表格（优势/劣势）

## 日志输出
- 调度日志: /tmp/monitor_dispatch.log
- 格式: `[时间] Bot状态 → 分配任务ID`

## 终止条件
- 到达指定时长
- 用户发送"停止调度"
